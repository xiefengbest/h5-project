# Pad 壁纸横竖屏内容策略

> 适用：TCL Pad 独立壁纸 App / 锁屏杂志 Pad 版  
> 状态：产品策略草案（内容尚未入库，供 PM / 设计 / 研发对齐）  
> 关联原型：`pages-pad/wallpaper/`（当前仅横屏 16:10 高保真）

---

## 1. 背景与待决问题

Pad 版壁纸内容尚未开始准备，以下问题需要先定策略再动内容生产与 API：

1. 横屏、竖屏是做 **一套内容** 还是 **两套内容**？
2. 每个壁纸是 **同一张图裁两套**，还是 **横竖完全不同的图**？
3. 用户 **旋转屏幕** 时，是否需要 **重新请求壁纸 feed 队列**？

---

## 2. 核心结论（可直接拍板）

| 问题 | 建议 |
|------|------|
| 横竖做一套还是两套？ | **一套内容（同一个 `wallpaperId`），两种展示规格** |
| 裁切还是不同图？ | **优先「同一源图预裁横竖两套」**；仅当无法裁切时才单独选图或单方向展示 |
| 旋转是否重新拉 feed？ | **否**。不刷新列表顺序与分页，只切换当前条目的展示 URL / 裁切资源 |
| 原型阶段准备什么？ | **先定数据模型（1 ID + 2 asset）**；视觉原型以横屏 16:10 为主即可 |

**一句话：不是两套内容队列，也不是一转屏就换一批壁纸；而是同一批内容，按方向换展示形态。**

---

## 3. 三种交付模式对比

### 模式 A：单图 + 客户端硬裁（不推荐作为主方案）

- 服务端只下发一张图，旋转时客户端 `object-fit: cover` 裁切。
- **优点**：内容只备一套，成本最低。
- **缺点**：主体易被裁掉（人脸、文案、 Logo），锁屏/杂志类对构图敏感，质量不稳定。
- **适用**：内部 Demo、对画质要求极低的场景。

### 模式 B：单源图 + 预裁横竖两套（推荐）

- 每个 `wallpaperId` 对应两个交付规格，例如：
  - `imageLandscape`：1280×800（16:10）
  - `imagePortrait`：800×1280（10:16）
- 入库时可从更大尺寸源图（如 2560×1600）按安全区自动裁切。
- **优点**：画质可控、feed 逻辑简单、旋转体验顺滑、运营仍是一套 ID。
- **缺点**：入库/审核多一道裁切流程（可自动化）。

### 模式 C：横竖完全两套图库 / 两套队列（默认不做）

- 横屏推风景、竖屏推人像，编辑策略完全分离，甚至独立 feed。
- **优点**：各方向构图最优。
- **缺点**：双倍选图/审核/运营；旋转时若换队列，用户感知为「内容突然变了」。
- **适用**：横竖内容类型差异极大，且产品明确接受「旋转 = 换一批内容」。

## 4. 旋转时的产品行为

### 4.1 用户预期

旋转设备时期望：

> 还是这些壁纸，只是换了个看法。

而不是：

> 换了个 App，换了一批内容。

### 4.2 推荐行为

```
用户旋转屏幕
  → 不刷新 feed 列表顺序
  → 不重新请求分页（已加载页）
  → 本地已加载 item 切换 landscape ↔ portrait 的 imageUrl
  → 若某条仅有单方向 asset：按 fallback 规则处理（见 §6）
  → （可选）下次冷启动 / 下拉刷新时，请求带 orientation 参数，仅影响排序权重
```

### 4.3 不推荐

- 旋转时整页 `reload` 或重新 `pull feed`
- 横竖使用两个独立 `feedId` / 两套推荐队列（除非走模式 C 且产品明确立项）

## 5. 内容数据模型（建议）

### 5.1 单条壁纸结构

```json
{
  "id": "wp_1001",
  "title": "Mountain Lake at Dawn",
  "type": "wallpaper",
  "assets": {
    "landscape": {
      "url": "https://cdn.example.com/wp_1001_l.jpg",
      "width": 1280,
      "height": 800,
      "aspectRatio": "16:10"
    },
    "portrait": {
      "url": "https://cdn.example.com/wp_1001_p.jpg",
      "width": 800,
      "height": 1280,
      "aspectRatio": "10:16"
    }
  },
  "orientationPolicy": "both",
  "sourceMaster": {
    "url": "https://cdn.example.com/wp_1001_master.jpg",
    "width": 2560,
    "height": 1600
  },
  "safeZone": { "x": 0.1, "y": 0.1, "w": 0.8, "h": 0.8 }
}
```

### 5.2 字段说明

| 字段 | 说明 |
|------|------|
| `id` | 全局唯一，横竖共用 |
| `assets.landscape` / `assets.portrait` | 面向客户端的交付 URL，建议 CDN 预裁好 |
| `orientationPolicy` | `both` \| `landscapeOnly` \| `portraitOnly` |
| `sourceMaster` | 可选，入库源图，便于重新生成裁切 |
| `safeZone` | 可选，自动裁切时的主体安全区（归一化坐标） |

### 5.3 列表 API 建议

**方案 1（推荐）：一次返回双 asset**

- 首屏列表响应内包含 `assets.landscape` 与 `assets.portrait`
- 旋转时 **零网络请求**，客户端本地切换 URL

**方案 2：按当前方向返回主 URL**

- 请求参数：`?orientation=landscape|portrait`
- 响应内主字段 `imageUrl` 对应当前方向
- 旋转时 **不重新拉 feed**，仅对可见卡片补请求缺失方向的 URL（或批量 prefetch 另一方向）

**共同原则：feed 的 item 顺序与 `id` 列表不因旋转而变。**

## 6. 入库规则：何时裁切、何时单独选图

| 素材类型 | 处理方式 | `orientationPolicy` |
|----------|----------|---------------------|
| 通用摄影 / 插画（主体居中、留有余量） | 一源双裁 | `both` |
| 强方向性构图（超宽全景、竖版人像） | 优先方向预裁；另一方向降级裁切或备选图 | `both` 或单方向 |
| 带固定横版/竖版文案的运营 Banner | 横竖各做一版设计稿（仍是同一 `id`，不是两套队列） | `both` |
| 完全无法裁切（横版文字在边缘等） | 仅单方向展示 | `landscapeOnly` 或 `portraitOnly` |

**原则：不是「每张都手工做两套图」，而是「每个 ID 在交付层有两个规格」——多数自动裁，少数人工或单方向。**

## 7. 规格与 Pad 原型对齐

| 场景 | 建议比例 | 参考像素 |
|------|----------|----------|
| Pad 横屏壁纸 | 16:10 | 1280×800 |
| Pad 竖屏壁纸 | 10:16 | 800×1280 |
| 手机壁纸（现有） | ~9:19.5 竖屏 | 390×844 量级 |

三者比例不同，**不能** 用手机竖图直接拉伸作为 Pad 横图，也不能假设 Pad 竖屏等于手机竖屏。

当前 H5 原型 `pages-pad/wallpaper/index.html` 已按 **横屏 16:10** 实现；竖屏资源可在数据模型预留，视觉稿后续补充。

---

## 8. 是否与「锁横屏」有关

| App 方向策略 | 内容侧要求 |
|--------------|------------|
| **锁定横屏** | 仅备 `assets.landscape` 即可；竖屏 asset 可不做 |
| **跟随系统旋转** | 必须预留 `assets.portrait`（或裁切 fallback） |
| **仅竖屏 Pad 场景** | 以 `portrait` 为主（与本项目 10 寸横屏主场景不同，需单独立项） |

**10 寸平板壁纸/杂志类常见做法：主场景横屏，但需保证旋转时不出现布局崩坏或主体被裁没。**

## 9. 决策树（团队拍板用）

```
Pad App 是否支持竖屏使用？
├─ 否（锁横屏）
│   └─ 只做 landscape asset，一套即可
└─ 是（可旋转）
    ├─ 旋转时 feed 是否变化？
    │   └─ 建议：否，同一队列、同一顺序
    └─ 每个 wallpaper 如何交付？
        ├─ 推荐：1 个 wallpaperId + 横/竖两个 URL（预裁，模式 B）
        └─ 不推荐默认：两套独立队列 / 两套无关图库（模式 C）
```

## 11. 与短剧等非壁纸内容的关系

| 内容类型 | 横竖策略 |
|----------|----------|
| **壁纸** | 本文策略：1 ID，横竖双 asset（16:10 / 10:16） |
| **短剧** | 内容本身偏竖屏 9:16，与壁纸策略独立；Pad 上可继续竖版海报，不必与壁纸横竖规则混为一谈 |
| **资讯** | 与方向弱相关，按 Pad 布局适配即可 |
